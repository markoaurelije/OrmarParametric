# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

An Autodesk Fusion 360 Add-In, written in Python against the `adsk` API (`adsk.core`, `adsk.fusion`, `adsk.cam`). It generates parametric wardrobe/cabinet furniture ("Ormari Parametric" — Croatian for "Parametric Wardrobes") by driving Fusion's user parameters, component visibility, and feature suppression from a custom command dialog.

There is no standalone runtime: `adsk` only exists inside Fusion 360's embedded Python interpreter. There is no `requirements.txt`, `pyproject.toml`, build step, linter config, or test suite in this repo — none of the usual Python tooling commands apply here.

## Running / debugging

- The add-in is loaded by pointing Fusion 360's Scripts and Add-Ins panel at this folder (identified by `OrmarParametric.manifest`, `id: 7f99ac39-252e-42db-93bd-e3903a22363e`).
- Once running inside Fusion, attach VS Code with the "Python: Attach" config in [.vscode/launch.json](.vscode/launch.json) (debugpy on `localhost:9000`) to set breakpoints and inspect state live.
- `config.DEBUG` (in [config.py](config.py)) controls whether log messages are mirrored to Fusion's Text Command window; errors always go there and to the Fusion log file regardless.
- There is no automated way to exercise this code outside a running Fusion session with the target document(s) open — changes should be verified manually in Fusion.

## Lifecycle & structure

- [OrmarParametric.py](OrmarParametric.py) is the add-in entry point Fusion calls: `run(context)` / `stop(context)`. It puts `lib/` on `sys.path` and delegates to `commands.start()` / `commands.stop()`.
- [commands/\_\_init\_\_.py](commands/__init__.py) is the registry: each subfolder under `commands/` is a self-contained command exposing `start()`/`stop()` in its `entry.py`, and must be added to the `commands` list here to be active. Currently only `commandDialog` is registered; `paletteShow`/`paletteSend` are Autodesk's stock sample-add-in templates (palette + postMessage bridge) kept around as reference/scaffolding but not wired in.
- [lib/fusionAddInUtils/](lib/fusionAddInUtils/) is Autodesk's standard add-in utility template: `general_utils.py` (`log`, `handle_error`) and `event_utils.py` (`add_handler`/`clear_handlers`, which wraps a callback in a generated `adsk` event-handler subclass and keeps a reference alive so it isn't GC'd). Treat this as boilerplate, not project-specific logic.
- [lib/openpyxl/](lib/openpyxl) and [lib/et_xmlfile/](lib/et_xmlfile) are vendored third-party packages (added because Fusion's Python has no package manager access). `openpyxl` is now used by [excel_export.py](commands/commandDialog/excel_export.py) to write the cut-list export into `narudzba-excel.xlsm` (an order-form template at the repo root).

## The `commandDialog` command (the actual product)

This is where all domain logic lives, under [commands/commandDialog/](commands/commandDialog/):

- **[dialog_config.py](commands/commandDialog/dialog_config.py)** is the single source of truth for the UI: a declarative list of `InputItem` (name, `InputType`, parent group, tooltip, min/max, table binding, etc.). Adding or changing a dialog field means editing this list — `utils.py` renders it generically, there's no per-field UI code to hand-write. Each `InputItem` (unless `input_has_no_param=True`, i.e. it's a pure layout group) maps 1:1 to a Fusion **user parameter** of the same name.
- **[utils.py](commands/commandDialog/utils.py)** does the heavy lifting:
  - `create_dialog`/`create_input` walk `input_items` and build the actual Fusion `CommandInputs` tree (groups, value/bool/integer/dropdown inputs, the ultrabox table and its toolbar buttons), seeding each from the matching user parameter's current expression/value.
  - `set_user_parameters_via_inputs` / `input_to_user_parameter` push edited dialog values back into Fusion user parameters on execute/preview.
  - `set_component_visibility` reads user parameters after they're set and toggles occurrence light bulbs (show/hide) and suppresses specific features (e.g. `"split ukrute"`, `"split police"`) based on flags like `pregrada` (divider) — this is how boolean toggles in the dialog become visible geometry changes.
  - **Prefix system**: multiple cabinets can coexist as parameter sets in one design, distinguished by a name prefix (e.g. `J1_sirina`, `O1_sirina`). `get_prefixes()` discovers all prefixes present by intersecting which prefixes exist for every non-group parameter name. Nearly every function threads a `prefix` argument through for this reason — the dialog renders one tab per prefix.
  - `add_parametric_component`/`rename_user_parameters` implement "add a new cabinet": copy the base design's root component into the target design and rename its `J1_*` parameters to `<newname>_*`.
- **[presets.py](commands/commandDialog/presets.py)** holds named parameter presets (`kuhinja_viseci_element`, `kuhinja_donji_element`, `komoda`) — full sets of `{paramName, expression}` applied together via `load_preset` in utils.py.
- **[ultrabox.py](commands/commandDialog/ultrabox.py)** is a work-in-progress feature for inserting drawer sub-assemblies ("Ultrabox") dynamically per cabinet, driven by add/remove toolbar buttons on the ultrabox table. The actual per-instance parameterization (step 3/4 in `perform_add_ultrabox`) is not yet implemented — only base copy-and-rename.
- **[event_handlers/](commands/commandDialog/event_handlers/)** wires Fusion's command lifecycle: `command_created_event_handler.py` builds the dialog, calls `clear_session_finish_overrides()`, and registers the other handlers; `input_changed_handler.py` reacts to preset selection, the "add cabinet" button, the ultrabox add/remove buttons, and the two finish pickers (`finish_paint_select`, `finish_band_select`); `command_execute_handler.py` / `command_execute_preview_handler.py` both run the same param-sync → visibility → **`apply_finish`** → ultrabox-materialization sequence (preview does it live as the user edits, execute commits on OK, and execute additionally calls `persist_finish_overrides()`) — keep these two handlers behaviorally in sync when changing this flow.

## Code-generated base cabinet

Cabinets are generated entirely from code by **[base_design.py](commands/commandDialog/base_design.py)** — the old workflow of copying a hand-modeled `J1` base document from the cloud is gone (`get_design_by_name`/`rename_user_parameters` in utils.py are leftovers of it). `add_cabinet(design, name)` creates a wrapper component named `name` at root and builds the full parametric cabinet inside it with `<name>_*` user parameters: every panel is an origin-anchored rectangle sketch + extrude driven by parameter expressions, positioned by a rigid joint to an expression-driven joint origin (equivalent to the original J1's face-to-face joints — geometry natively follows parameter edits). The declarative tables at the top of the module (`USER_PARAMS`, `PANELS`, `UKRUTA_POSITIONS`, `ULTRABOX_PANELS`) were reverse-engineered from the original J1 design and are the single source of truth for cabinet geometry.

Two Fusion API gotchas are baked into `create_cabinet` and must be preserved: (1) new occurrences default to `isGroundToParent=True`, which silently breaks joints until unset; (2) rectangular patterns and cross-component combine features created *inside* a nested component ignore/fail on other occurrences' joint-driven transforms — they must be created on `design.rootComponent.features` using root-context proxies (`createForAssemblyContext`). Component/feature names are load-bearing: `utils.set_component_visibility` matches occurrences by name prefix (`"polica"`, `"ukrute"` wrapper with children, features `"split police"`/`"split ukrute"`) and finds each cabinet as a root occurrence whose component name equals the parameter prefix minus the underscore; `ultrabox.py` needs a component named `"Ultrabox"`.

A third gotcha governs `utils.add_parametric_component` specifically: while our command dialog is active on a document, Fusion silently rolls back any *structural* model changes (new occurrences/bodies) made from the `inputChanged` handler on the very next preview cycle — only parameter edits survive, because `set_user_parameters_via_inputs` re-applies them every cycle, whereas a one-shot component creation is never re-applied and gets discarded. A cabinet built by `add_cabinet` directly into the document the dialog is open on will flash into existence and vanish before the user can see it. `add_parametric_component` works around this by always calling `app.documents.add(...)` and building the cabinet into that brand-new, command-free document instead — never into the currently active one. This is also why the pre-code-gen implementation always opened a separate "target design" rather than inserting into the active document; don't remove that behavior when touching this function.

## Board finish & edge banding (kantiranje)

Every board is painted so the model reads like the finished cabinet, and this data feeds the cut-list export (see below). Lives mostly in `utils.py` (`apply_finish` and helpers) plus rule tables in `base_design.py`.

- **Each board carries a *decor* (a named colour), and each edge carries its own band colour.** *Decor*: the board's broad faces are painted the decor's colour; a board defaults to the palette's `default_front` decor if it's "coloured" (visible outside) or `default_interior` (white) otherwise. Different boards can carry different decors → single-colour or two-tone/accent, same machinery. *Edge banding*: each of the four narrow (thickness) edges is either banded **in some decor colour** (independent of the board's colour — e.g. a black band on a white board) or raw (a striped "no banding" texture, so a raw edge is never confused with a board face — including white-on-white). Rule-banded edges default to the board's own decor (matching band); clicking an edge bands it in the active decor. Only visible edges should be banded.
- **The decor palette lives in [decors.json](commands/commandDialog/decors.json)** — `decors` (each `{name, rgb, code}`; `rgb` is only the 3D preview, `name`/`code` feed the future order/cut-list export) plus `default_front` / `default_interior`. Loaded by `_load_decors()` in `utils.py` (small built-in fallback if missing). `decor_rgb(name)` gives the preview colour.
- **Which boards are coloured-by-default and which edges are banded live in [board_rules.json](commands/commandDialog/board_rules.json)** (schema in [board_rules.README.md](commands/commandDialog/board_rules.README.md)). Per board: `colored` (bool/condition — maps a default board to `default_front` vs `default_interior`) and `banding` (map of edge orientation → bool/condition). `base_design.py` holds the same as fallback defaults (`COLORED_DEFAULTS`, the `banding` dicts in `PANELS`) and `_load_board_rules_json()` overlays the JSON on top. Orientations are world-frame: `front` +Y, `back` −Y, `left` +X, `right` −X, `top` +Z, `bottom` −Z. A condition string is Python evaluated against the cabinet's flag parameters by base name (e.g. `"bokovi_preko_donje_ploce == 0"`), re-evaluated every preview so it follows the construction. Defaults: coloured = doors + top panel; banding = the visible edges per board. `base_design.board_banding_counts(name, flags, dims)` reduces a board's **rule-default** banding to the `(long, short)` counts, and `base_design.banding_counts_for_orientations(name, orientations, dims)` does the same from an **explicit set of banded orientations** — the cut-list export uses the latter fed by `effective_banding` so its counts include the user's manual banding overrides (see below), not just the rule defaults.
- **`apply_finish(prefix)`** (called after params are set, every preview and execute, next to `set_component_visibility`) walks the cabinet **from its root-level wrapper occurrence via `childOccurrences`** and paints each board body. It must not use a nested component's `allOccurrences` — those proxies lack the root→wrapper path and `BRepFace.appearance` then raises `InternalValidationError` (painting silently no-ops). Faces are classified by `_classify_face` (bounding-box outer plane + thinnest axis = thickness): the two broad faces vs the four narrow edges, robust to the back-panel groove and divider split cuts (recessed faces are skipped). One solid appearance is created per decor on demand (`"Ormar - <decor>"`), plus `"Ormar - bez kanta"` (the stripe — copied from the library appearance **"Fabric (Grey)"**, whose Colour already has a connected image texture, then `changeTextureImage(resources/textures/no_banding.png)`; just flipping `hasConnectedTexture` does not create a texture). The appearance library's name varies by Fusion version, so it's found by matching `"Appearance"` in the library name.
- **User overrides via a decor dropdown + two click pickers.** The `finish_active_decor` dropdown ("Boja", populated from the palette) chooses the *active decor* used by **both** pickers. `finish_paint_select` ("Oboji ploču") → `set_board_decor(entity, active_decor)` paints a whole board that decor (pick "Bijela" to send it back to interior white); `finish_band_select` ("Kantiraj rub") → `set_edge_band(entity, active_decor)` bands the clicked edge (orientation via `_classify_face`) in the active decor — re-clicking an edge already banded in that same decor turns it raw. Both pickers are `SelectionCommandInput`, `SolidFaces` filter, momentary (the click applies and the selection is cleared). The cabinet a pick belongs to is read from `occ.fullPathName.split("+")[0].split(":")[0]`.
- **The inputChanged rollback gotcha applies here too** (see `add_parametric_component` above): writing an override *attribute* from the `inputChanged` handler is rolled back on the next preview, so clicks appeared to do nothing. The fix mirrors the param-sync pattern — clicks write **in-memory session dicts** (`_session_decor_overrides` keyed `(prefix, spec)` → decor name, `_session_banding_overrides` keyed `(prefix, spec, orient)` → decor name or `""` for raw) which `apply_finish` re-reads every cycle via `effective_decor` / `effective_banding` (returns `{orient: decor}`; priority: session → persisted attribute → rule default, rule-banded edges defaulting to the board decor). `persist_finish_overrides()` flushes the session to occurrence attributes **only on execute/OK** (groups `OrmarFinish` = `spec` → decor name, `OrmarBanding` = `spec:orient` → decor name / `""`; legacy `"1"/"0"` values still map to default front/interior decor and banded-in-board-decor / raw); `clear_session_finish_overrides()` resets the session when the dialog reopens so a cancelled session doesn't leak.

## Cut-list Excel export (krojna lista)

[excel_export.py](commands/commandDialog/excel_export.py) writes every cabinet's board cut list into a copy of the Elgrad supplier order form `narudzba-excel.xlsm`. Triggered by the **`exportCutListButton`** ("Izvezi krojnu listu u Excel") dialog button, handled in [input_changed_handler.py](commands/commandDialog/event_handlers/input_changed_handler.py). Pure reads only — it never edits the model, so it's independent of the preview/rollback lifecycle.

- **Only *visible* boards are exported.** `_gather_rows` walks each cabinet's occurrence tree (`_collect_boards`, same tree `apply_finish` walks) gathering `(spec, body)` pairs; an occurrence whose **light bulb is off hides its whole subtree** (this is how the `ukrute`/`Ultrabox` wrappers and flag-driven panel hides work), so hidden components — manual or flag-driven — contribute nothing. Each solid **body** is one physical piece, so a shelf split in two by a divider becomes two pieces; each materialized `Ultrabox N` drawer contributes its child boards while the hidden `Ultrabox` template is skipped. Identical measured pieces `(spec, DULJINA, ŠIRINA)` collapse into one row with a quantity.
- **Dimensions are measured from the real 3D geometry, not the parametric `size` expressions** — so a user's manual edits to a board after creation are reflected. `_board_geometry_dims` uses the **greatest-area-faces** method: a board's two broad faces are its largest, so the largest face's normal is the thickness axis; length/width are the extent of the body's vertices projected onto the board's own in-plane axes (`_dims_from_geometry`). Measuring in the board's *own* frame makes it rotation-invariant — a door swung open on its revolute joint still measures correctly, where an axis-aligned bounding box would not — and robust to grooves/split cuts (which only cut inward). `world_dims` (each own-axis extent assigned to its nearest world axis) feeds the banding long/short reduction.
- **Banding counts match the model.** For each board the export reads `utils.effective_banding(...)` (rule defaults **plus** the user's session/persisted banding overrides) and feeds the banded orientations to `base_design.banding_counts_for_orientations` — so the cut list can never diverge from what `apply_finish` paints. Written to the ABS-2mm columns (G/H) only; the mini/mel columns (E/F) stay blank (single banding type). Zero counts are written blank, matching the form's own `PRIMJER` sheet.
- **Column map (`Elementi` sheet, data from row 4; row A's R.B. numbering left intact):** B DULJINA, C ŠIRINA, D BR.KOM., G ABS-long, H ABS-short, I NAPOMENA (`"<cabinet> <board>"`, e.g. `"J1 bok desno"`). Row order: bottom → sides → back → top → divider → shelves → plinth → doors → stiffeners → ultrabox. `_write_to_excel` loads the template with `keep_vba=True`, clears B–I of rows 4–102, writes, and saves a **copy** (`<docname>-krojna-lista.xlsm` on the Desktop, fallback home) — the template is never overwritten.

| ploca | panel/board | cokla | plinth/kickboard |
